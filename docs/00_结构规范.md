# 工程结构规范

本规范用于统一目录分层与命名，后续新增或迁移请按此执行。

## 1. 顶层目录约定
- `apps/`：应用层（前端/客户端/可视化工具）
- `services/`：服务层（后端/网关/接口/后台）
- `firmware/`：固件与硬件相关工程
- `packages/`：SDK/共享库/组件包
- `tools/`：本地工具与双击入口
- `scripts/`：统一运维脚本
- `docs/`：文档
- `.cache/`：构建/日志/输出/备份/临时

## 2. 命名规范
- 目录统一使用 `kebab-case`，例如：`ai-pet-provisioner`
- 文件保持原约定（如已有中文脚本名可保留）
- 新增项目不要使用数字或单字目录名

## 3. 缓存与产物
- 所有构建产物、日志、输出、备份统一放到 `.cache/`
- 推荐结构：`.cache/{build,logs,dist,output,backups,tmp,legacy}`
- 脚本默认只写 `.cache/logs` 与 `.cache/dist`

## 4. 路径引用原则
- 文档路径引用以根目录为起点
- 脚本路径引用统一使用 `$ROOT_DIR/...`
- 如需写绝对路径，需与当前目录结构一致

## 5. 特例说明
- Flutter 项目 `pubspec.yaml` 的 `name` 不能使用 `-`，保持 `ai_pet_provisioner` 合规
- 目录名已规范为 `apps/ai-pet-provisioner`，不与 Flutter 包名绑定
