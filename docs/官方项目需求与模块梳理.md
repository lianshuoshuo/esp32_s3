# 官方项目主线梳理（小智）

## 1. 目标（回到官方主线）
从现在开始，优先以官方仓库为核心开展工作，原则是：

- 先理解官方模块，再做定制改动
- 优先“最小改动”，避免偏离官方升级路径
- 自研脚本/UI 只做运维增强，不替代官方业务逻辑

## 2. 当前工程分层
本仓库可以分成 3 层：

1. 官方服务端层：`services/xiaozhi-esp32-server/`
2. 官方固件层：`firmware/xiaozhi-esp32/`
3. 本地运维增强层：`scripts/`、`apps/visual-command-ui/`、`tools/mac/`

## 3. 官方服务端模块图（xiaozhi-esp32-server）
关键目录：

- `main/xiaozhi-server/`：Python 核心服务（设备通信、会话处理、ASR/LLM/TTS/VAD、工具调用、MCP）
- `main/manager-api/`：Java 管理接口（后台业务 API）
- `main/manager-web/`：Web 管理端前端
- `main/manager-mobile/`：移动端管理前端

Python 核心服务重点子模块：

- `core/websocket_server.py`：WebSocket 接入
- `core/http_server.py`：HTTP 接口
- `core/handle/`：消息处理与分发
- `core/providers/`：能力提供者抽象（ASR/LLM/TTS/VAD/Memory/Tools）
- `core/providers/tools/`：IOT/MCP/插件工具调用链

## 4. 官方固件模块图（xiaozhi-esp32）
关键目录：

- `main/application.cc`：应用主流程
- `main/protocols/`：通信协议（WebSocket / MQTT）
- `main/audio/`：采集、编解码、音频处理
- `main/display/`：屏幕显示
- `main/mcp_server.*`：设备侧 MCP 工具注册与调用
- `main/boards/`：板级适配（你当前关注 `boards/otto-robot`）

`otto-robot` 下与动作控制相关：

- `otto_controller.cc`
- `otto_movements.cc`
- `oscillator.cc`

已完成：

- 本地官方服务端全链路可运行
- 本地 MQTT 网关可运行
- 本地 MCP 接入点服务可运行（端口 8004）
- IP 同步、启动停止、诊断、日志等运维脚本
- 可视化运维控制台与 DMG 打包能力
- 官方模块文档（服务端/网关/固件/板级/MCP路径）已整理
- MCP 双路径已配置：
  - 路径 B：`data/.mcp_server_settings.json`
  - 路径 C：`server.mcp_endpoint` + `mcp-endpoint-server`

未完成（官方主线视角）：
- 设备在线时触发一次完整 MCP 工具调用，补齐“运行期日志验收截图”
- 根据真实使用场景继续扩展 MCP 工具（目前已具备最小落地能力）

## 6. 接下来建议执行顺序
建议按以下顺序推进（先服务端后固件）：

1. 服务端链路梳理：`main/xiaozhi-server` 的入口与消息流
2. 网关链路梳理：`xiaozhi-mqtt-gateway` 的 MCP 指令收发流
3. 固件链路梳理：`firmware/xiaozhi-esp32/main/protocols` + `main/mcp_server.*`
4. 板级梳理：`boards/otto-robot` 的设备能力与可裁剪项
5. 最后再做定制改动，且每次改动都对应官方模块定位

## 7. 本地协作约定
- 任何业务改动先标注“改的是官方哪一层模块”
- 优先提交“阅读结论 + 改动原因”再动代码
- 尽量把定制逻辑放在运维层，不侵入官方核心路径
