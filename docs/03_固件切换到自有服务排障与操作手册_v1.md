# 固件切换到自有服务排障与操作手册（v1）

更新时间：2026-03-19

## 1. 先看结论
- 当前这套本地链路已经验证通过：`官方服务端 + 官方网关 + 自定义固件 OTA 地址` 可以正常绑定到我们自己的服务。
- 本轮最关键的问题，不是后台服务本身，而是**设备固件实际编译进去的 OTA 地址仍然指向官方**。
- 另一个会放大问题的点是：设备会把 `ota_url / mqtt.endpoint / websocket.url` 缓存在 NVS 里，所以只改服务端、不清缓存，设备仍可能继续显示 `xiaozhi.com`。

一句话记忆：

> 设备连谁，先看固件里实际编译进去的 `CONFIG_OTA_URL`，再看设备 NVS 里有没有旧缓存，最后才看服务端参数。

---

## 2. 这次问题是怎么定位出来的

### 2.1 现象
- 我们自己的固件刷进去后，设备屏幕和播报仍显示 `xiaozhi.com`
- Web 后台使用激活码绑定时，经常提示“激活码错误”
- 但使用**官方固件**，并在 Wi-Fi 配网时手动填入我们的自定义 OTA 地址后，绑定又是正常的

### 2.2 推理过程
这说明：
- 我们自己的 `xiaozhi-esp32-server + xiaozhi-mqtt-gateway + manager-api` 并不是完全坏的
- 问题更像是：**我们的固件没有真正走到我们自己的 OTA / websocket / mqtt 链路**

### 2.3 最终确认的根因
真正的根因有 3 层：

1. 固件**实际编译生效**的 `CONFIG_OTA_URL` 仍然是官方地址  
   旧值：
   - `https://api.tenclass.net/xiaozhi/ota/`

2. 我们最早改的本地地址路径也不完整  
   错误写法：
   - `http://192.168.77.221:8002/ota/`
   正确写法：
   - `http://192.168.77.221:8002/xiaozhi/ota/`

3. 设备会把服务地址缓存进 NVS  
   即便服务端已经改对，只要设备里缓存的还是旧地址，它仍会继续连旧服务。

### 6.2 当设备仍显示 `xiaozhi.com` 时
按这个顺序排查，不要跳步：

1. 检查服务端参数是不是当前 IP
2. 检查固件源码里的 `CONFIG_OTA_URL`
3. 检查构建产物里的 `CONFIG_OTA_URL`
4. 清空整片 Flash 并重刷
5. 重新配网
6. 看屏幕和播报是否还显示 `xiaozhi.com`

### 6.3 当官方固件 + 自定义 OTA 能正常，而我们固件不正常时
这个场景优先怀疑：
- 不是服务端
- 而是我们自己固件的编译配置或 NVS 缓存

## 8. 后续最容易再次踩坑的点

### 坑 1：只改 `Kconfig.projbuild`，不改 `sdkconfig`
现象：
- 你以为已经改了 OTA 默认值
- 实际构建产物仍然是旧地址

解决：
- 一定同时检查 `sdkconfig`
- 编译后再检查 `build_esp32s3_v552/config/sdkconfig.h`

### 坑 2：只重启服务，不重刷设备
现象：
- 服务端参数对了
- 设备仍显示旧域名

解决：
- 因为设备里还有旧 `ota_url / mqtt / websocket` 缓存
- 要清空并重刷，或者至少让新固件的自愈逻辑真正跑起来

### 坑 3：OTA 路径少了 `/xiaozhi`
错误写法：
- `http://IP:8002/ota/`

正确写法：
- `http://IP:8002/xiaozhi/ota/`
