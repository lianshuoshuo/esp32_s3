# 换 IP 与换云服务器迁移清单（v1）

更新时间：2026-03-19

## 1. 这份文档解决什么问题
- 当本机局域网 IP 变化时，哪些地方必须一起改
- 当后续从“本机联调”切到“云服务器部署”时，哪些配置必须同步切走
- 如何避免再次出现：
  - 设备仍显示 `xiaozhi.com`
  - 设备能连通但绑定失败
  - 服务端已改好，但固件仍连旧地址

一句话记忆：

> 换环境不是只改一个地址，而是要同时改：服务端参数、固件 OTA 默认值、设备缓存、脚本默认值。

---

## 2. 当前这套链路里，最关键的 4 类地址

| 类别 | 作用 | 当前本机示例 |
|---|---|---|
| 前端地址 `server.fronted_url` | 设备屏幕/播报里的绑定地址来源 | `http://192.168.77.221:8002/` |
| OTA 地址 `server.ota` / `CONFIG_OTA_URL` | 设备启动后先访问的配置入口 | `http://192.168.77.221:8002/xiaozhi/ota/` |
| WebSocket 地址 `server.websocket` | 设备主会话通道 | `ws://192.168.77.221:8000/xiaozhi/v1/` |
| MQTT / UDP / 状态 API | 网关链路与状态检查 | `192.168.77.221:1883` / `192.168.77.221:8884` / `192.168.77.221:8007` |

其中：
- **设备先看 OTA**
- **OTA 再下发 websocket / mqtt**
- **绑定页面显示给用户看的地址，主要看 `server.fronted_url`**

---

## 3. 什么时候必须执行迁移清单

出现下面任意一种情况，就不要只重启服务，要按完整清单执行：

1. Mac 换了 Wi-Fi，局域网 IP 变化
2. 准备从本机调试切到云服务器
3. 设备屏幕或播报又出现 `xiaozhi.com`
4. 官方固件填自定义 OTA 正常，但我们自己的固件异常
5. Web 后台绑定码提示“激活码错误”

## 4. 本机 IP 变化时的最小清单

### 4.1 先确认新 IP
```bash
ipconfig getifaddr en0 || ipconfig getifaddr en1
```

### 4.2 同步服务端参数
```bash
cd /Users/lss/Desktop/AI_MCP
./scripts/同步本机IP.sh
./scripts/重启本地服务.sh
./scripts/查看服务状态.sh
```

至少确认这些值已经变成新 IP：
- `server.fronted_url`
- `server.ota`
- `server.websocket`
- `server.mqtt_gateway`
- `server.udp_gateway`
- `server.mqtt_manager_api`
- `server.mcp_endpoint`

### 4.3 如果设备已经刷过旧固件
如果只是服务端 IP 变了，但设备上跑的固件编译进去的 `CONFIG_OTA_URL` 还是旧地址，那么设备仍可能跑到旧链路。

需要检查：
```bash
rg -n "CONFIG_OTA_URL" \
  /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/sdkconfig \
  /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/main/Kconfig.projbuild \
  /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/build_esp32s3_v552/config/sdkconfig.h
```

### 4.4 必要时重刷设备
如果下面任意一项不对，就直接清空并重刷：
- `CONFIG_OTA_URL` 仍是旧 IP
- 设备屏幕仍显示旧域名
- 设备没有重新走到我们的 OTA

```bash
cd /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32
bash /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/命令脚本/清空并重刷.command --yes
```

## 5. 切到云服务器时的完整清单

假设后续云端域名是：
- 管理页：`https://pet.example.com`
- OTA：`https://pet.example.com/xiaozhi/ota/`
- WebSocket：`wss://pet.example.com/xiaozhi/v1/`
- MQTT：`mqtt.pet.example.com:1883`

### 5.1 服务端参数要改
最少要同步这几个：

| 参数 | 云端示例 |
|---|---|
| `server.fronted_url` | `https://pet.example.com` |
| `server.ota` | `https://pet.example.com/xiaozhi/ota/` |
| `server.websocket` | `wss://pet.example.com/xiaozhi/v1/` |
| `server.mqtt_gateway` | `mqtt.pet.example.com:1883` |
| `server.udp_gateway` | `pet.example.com:8884` |
| `server.mqtt_manager_api` | `pet.example.com:8007` |
| `server.mcp_endpoint` | 按部署结果填写 |

### 5.2 固件默认值要改
至少同步这 2 处：
- `/Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/sdkconfig`
- `/Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/main/Kconfig.projbuild`

需要让 `CONFIG_OTA_URL` 指向云端 OTA，例如：
```text
https://pet.example.com/xiaozhi/ota/
```

如果你要保留板级默认值一致性，还要一起改：
- `/Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/main/boards/waveshare/esp32-s3-touch-lcd-4.3c/sdkconfig.4_3c`

### 5.3 重新编译并确认构建产物
```bash
cd /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32
bash /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/命令脚本/一键编译.command
rg -n "CONFIG_OTA_URL" \
  /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/build_esp32s3_v552/config/sdkconfig.h \
  /Users/lss/Desktop/AI_MCP/firmware/xiaozhi-esp32/build_esp32s3_v552/config/sdkconfig.cmake
```

### 5.4 重刷设备
云端切换时，建议直接清空并重刷，不要赌旧缓存不会干扰。

## 8. 最容易遗漏的坑

### 坑 1：只改服务端，不改固件
结果：
- 官方固件自定义 OTA 正常
- 我们自己的固件继续跑旧地址

### 坑 2：只改源码，不看构建产物
结果：
- 你以为改了
- 实际编译进去的 `CONFIG_OTA_URL` 还是旧值

### 坑 3：只重刷，不清空
结果：
- 设备里的 `ota_url / mqtt / websocket` 旧缓存继续干扰

### 坑 4：OTA 路径漏了 `/xiaozhi`
错误：
- `/ota/`

正确：
- `/xiaozhi/ota/`

### 坑 5：只看源码里有没有 `xiaozhi.com`
说明：
- 二进制里出现 `xiaozhi.com / xiaozhi.me / api.tenclass.net`，不一定代表还在连官方
- 有些字符串只是保留给“旧云端地址识别与自愈”逻辑使用
- 真正要看的是：
  - 构建产物里的 `CONFIG_OTA_URL`
  - 联网后的串口日志

## 9. 建议的长期做法

### 9.1 本机调试
- 用脚本同步当前 IP
- 本机 OTA / websocket / mqtt 全部统一成当前局域网 IP

### 9.2 云端部署
- 统一走域名，不要再用裸 IP
- WebSocket 改成 `wss://`
- OTA 改成正式 HTTPS
- MQTT 建议独立子域名

### 9.3 固件策略
- 本地联调固件：可保留本机默认 OTA
- 发布固件：建议切到正式云端域名
- 两套配置最好分开管理，不要混编
